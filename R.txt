Документация к проекту «Сервер картинок 2.0»

Данный проект представляет собой легковесный, отказоустойчивый и производительный веб-сервис для загрузки, хранения, постраничного просмотра и удаления изображений.
Проект разработан на чистом Python без использования сторонних веб-фреймворков и упакован в Docker-контейнеры с архитектурным разделением обязанностей между Nginx, Python-бэкендом и СУБД PostgreSQL.


1. Требования к окружению (Environment Requirements)

2. Для успешного запуска и эксплуатации сервиса на целевой машине должны быть установлены следующие компоненты:
1) Операционная система: Windows 10/11 (Pro, Enterprise или Home с WSL2), macOS или Linux (Ubuntu/Debian/CentOS).
2) Среда контейнеризации: Docker Desktop (версии 4.x и выше) — включает в себя Docker Engine и Docker Compose.
        Для Linux-серверов: Docker Engine (версии 20.10+) и плагин Docker Compose v2 (версии 2.x+).
3) Аппаратные ресурсы (минимальные):Оперативная память:
        2 ГБ свободного ОЗУ.
        Дисковое пространство: минимум 500 МБ свободного места на диске (зависит от объема загружаемых картинок и бэкапов).
4) Сетевые настройки: Свободный порт 80 на хост-машине (используется Nginx для приема входящего трафика).


2. Команды для управления проектом (Запуск и Остановка)

Управление жизненным циклом приложения осуществляется из терминала (Командной строки, PowerShell или терминала Linux/macOS) из корневой папки проекта:
        Первичный запуск и сборка проекта:
            Bash
            docker compose up --build
            Эта команда скачает базовые образы, соберет кастомный образ бэкенда, инициализирует базу данных, создаст необходимые таблицы, запустит фоновый планировщик бэкапов и поднимет прокси-сервер.

        Обычный запуск (без пересборки образов):
            Bash
            docker compose up -d
            Флаг -d запускает контейнеры в фоновом (демон) режиме, освобождая консоль для дальнейшей работы.
        
        Просмотр логов в реальном времени:
            Bash
            docker compose logs -f

        Остановка сервиса с сохранением всех данных:
            Bash
            docker compose down
            Контейнеры останавливаются и удаляются, но пользовательские картинки, логи и данные в PostgreSQL остаются в безопасности благодаря именованным томам (Volumes).

        Полная очистка (остановка контейнеров с удалением томов данных):
            Bash
            docker compose down -v
            Внимание: эта команда полностью сотрет базу данных и загруженные файлы!

4. Описание маршрутов (API Endpoints) и их функциональности

Вся маршрутизация входящих запросов разделена на два уровня: Nginx (маршрутизация статики на диск) и Python-бэкенд (маршрутизация динамических операций).

Сводная таблица маршрутов:
Маршрут             Метод       Компонент-обработчик        Назначение                                  Формат ответа
/upload             POST        Python Бэкенд               Загрузка нового изображения на сервер       JSON
/images/<filename>  GET         Nginx (напрямую с диска)    Быстрая отдача физического файла картинки   Бинарный (изображение)
/list               GET         Python Бэкенд               Просмотр списка картинок с пагинацией       HTML
/delete             GET         Python Бэкенд               Удаление картинки по её идентификатору      HTTP 303 (Перенаправление)

Подробное описание логики работы маршрутов:
1. Загрузка файлов (POST /upload)
    Функциональность: Принимает бинарные данные формы формата multipart/form-data.
    Логика бэкенда:
        1. Проверяет заголовок Content-Length. Если размер тела превышает 5 МБ, возвращает ошибку 400 Bad Request.
        2. Парсит байтовый поток, извлекает оригинальное имя файла и его расширение.
        3. Проверяет расширение по белому списку (.jpg, .jpeg, .png, .gif). Запрещенные форматы отклоняются с ошибкой 400.
        4. Генерирует уникальное имя файла с помощью алгоритма UUIDv4.
        5. Сохраняет файл в папку /images.
        6. Делает запись в PostgreSQL, сохраняя: уникальный ID, новое имя, оригинальное имя, размер в байтах, формат файла и дату/время загрузки.
    Пример успешного ответа (JSON, Статус 201):
        JSON
        { "url": "/images/a4f2b1c8d9e041b2a3c4d5e6f7a8b9c0.png" }


2. Раздача изображений (GET /images/<имя_файла>)
    Функциональность: Самый частый запрос в системе. Обрабатывается исключительно Nginx, минуя медленный интерпретатор Python.
    Логика работы:  Nginx сопоставляет URL-адрес с локальным хранилищем /images/, подставляет правильный MIME-тип из системы и отправляет файл клиенту.
                    Дополнительно проставляются заголовки кэширования Cache-Control со сроком жизни 30 дней для оптимизации нагрузки.

3. Веб-интерфейс списка (GET /list)
    Функциональность: Отображает динамическую HTML-страницу со всеми картинками, хранящимися в системе.
    Пагинация (Постраничный вывод): Маршрут принимает необязательный query-параметр ?page=.
    По умолчанию открывается page=1. На чистом SQL-запросе к PostgreSQL выполняется выборка порциями по 10 штук с конца:
        SQL
        SELECT id, filename, original_name, size, upload_time 
        FROM images ORDER BY upload_time DESC LIMIT 10 OFFSET <смещение>
    Интерфейс автоматически рассчитывает общее количество записей.
    Если пользователь находится на первой странице, кнопка «Назад» блокируется.
    Если элементы для следующей страницы отсутствуют, блокируется кнопка «Вперед».

4. Удаление изображений (GET /delete?id=<идентификатор>)
    Функциональность: Удаляет сущность из системы.
    Логика бэкенда:
        1) Извлекает параметр id из строки запроса.
        2) Делает запрос к БД, чтобы узнать физическое имя файла, привязанное к этому ID.
        3) Удаляет запись из таблицы images в рамках транзакции.
        4) При успешном удалении из БД вызывает системную функцию os.remove() для физического удаления файла с диска в папке /images.
        5) Отправляет клиенту статус 303 See Other с заголовком Location: /list, благодаря чему браузер пользователя мгновенно перенаправляется обратно на обновленный список картинок.

5. Документация по Резервному Копированию (Backup Guide)
Безопасность данных обеспечивается за счет автоматического изолированного резервного копирования метаданных из PostgreSQL.

Архитектура бэкапов
В бэкенд интегрирован автономный фоновый поток (Thread-демон), который активируется сразу после успешной инициализации сервера.
Он работает параллельно с основным веб-сервером и не блокирует обработку входящих HTTP-запросов от пользователей.
Поток выполняет резервное копирование с заданным интервалом (по умолчанию в коде настроено демонстрационное значение раз в 5 минут, для продакшена выставляется 86400 секунд — 24 часа).

Формат имен файлов
Каждый бэкап сохраняется с уникальной меткой времени в строгом соответствии с ISO-форматом:
    Plaintext
    backup_YYYY-MM-DD_HHMMSS.sql

Пример: backup_2026-05-30_195823.sql — бэкап, созданный 30 мая 2026 года в 19:58:23.

Все файлы автоматически записываются в директорию /backups, которая примонтирована к реальной папке на жестком диске хост-компьютера.
Это гарантирует сохранность бэкапов даже при полном уничтожении Docker-контейнеров.

Инструкция по ручному созданию бэкапа
Если вам необходимо экстренно сделать бэкап перед проведением технических работ, выполните в терминале хоста команду:

    Bash
    docker exec -t postgres_container pg_dump -U postgres images_db > backups/backup_manual.sql

Инструкция по восстановлению базы данных из бэкапа
В случае системного сбоя, повреждения таблиц или случайного удаления данных, вы можете восстановить состояние базы данных из любого имеющегося .sql файла бэкапа.
1) Перейдите в терминале в корневую папку проекта image-server.
2) Выберите из папки backups нужный файл бэкапа (например, backup_2026-05-30_175500.sql).
3) Выполните команду перенаправления потока ввода в контейнер базы данных:

    Bash
    docker exec -i postgres_container psql -U postgres images_db < backups/backup_2026-05-30_175500.sql

5) Утилита psql автоматически выполнит все SQL-команды из файла: воссоздаст структуру таблицы images и заполнит её историческими записями метаданных.
Веб-страница /list сразу же станет актуальной.
