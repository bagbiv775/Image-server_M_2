Инструкция по развертыванию и техническая документация проекта
Данный документ содержит руководство по развертыванию, описание архитектуры, маршрутизации и процессов резервного копирования веб-приложения для загрузки и просмотра изображений в соответствии с требованиями ТЗ 1.0 и ТЗ 2.0.

1. Требования к окружению
Проект полностью контейнеризирован с помощью Docker, что исключает необходимость ручной установки зависимостей (таких как Python, PostgreSQL или Nginx) на хост-машину.
Минимальные системные требования:
•	Операционная система: Windows 10/11 (с компонентом WSL2), macOS или Linux.
•	Установленное ПО:
o	Docker Desktop (версии 20.10.0 или выше)
o	Docker Compose (версии 2.0.0 или выше)
•	Свободные порты: Порт 80 (используется Nginx для входящего трафика). Убедитесь, что локальные веб-серверы (например, Apache, IIS или другие экземпляры Nginx) остановлены.
•	Сеть: Стабильное интернет-соединение для первой сборки образов.

2. Команды для запуска и остановки проекта
Все команды выполняются из корневой директории проекта через терминал (или встроенный терминал PyCharm).
Первый запуск / Пересборка с очисткой кэша (после изменений)
Если вносились изменения в конфигурационные файлы (Dockerfile, nginx.conf, шаблоны), запускайте проект с принудительным игнорированием кэша:
Bash
docker compose down
docker compose up --build --force-recreate -d
Стандартный запуск проекта (в фоновом режиме)
Bash
docker compose up -d
Остановка проекта (без удаления данных)
Команда останавливает контейнеры, сохраняя все файлы на жестком диске и данные в PostgreSQL:
Bash
docker compose stop
Полная остановка с уничтожением контейнеров
Удаляет контейнеры и внутренние виртуальные сети (данные на смонтированных томах volumes не удаляются):
Bash
docker compose down
Просмотр логов в реальном времени
Bash
docker compose logs -f

3. Описание структуры проекта
Проект организован по модульному принципу. Логика бэкенда, настройки веб-сервера и резервные копии разделены по изолированным директориям.
Plaintext
├── app/
│   ├── main.py              # Основной бэкенд-сервер на чистом Python (BaseHTTPRequestHandler)
│   ├── list_template.html   # HTML/CSS/JS шаблон интерфейса (разделение представления и логики)
│   └── Dockerfile           # Инструкция сборки контейнера Python-приложения
│   └── requirements.txt     # Файл с указанием внешних библиотек
├── nginx/
│   └── nginx.conf           # Конфигурационный файл Nginx (маршрутизация и раздача статики)
├── backup/                  # Локальная папка на хосте для хранения файлов резервных копий (.sql)
├── images/                  # Локальная папка на хосте для физического хранения загруженных картинок
├── docker-compose.yml       # Скрипт оркестрации и связи всех сервисов (бэкенд, БД, прокси, бэкапер)
└── README.md                # Текущая документация проекта

4. Подробное описание маршрутов и их функциональности
Вся маршрутизация входящих запросов разделена между веб-сервером Nginx (работает как Reverse Proxy) и бэкендом на Python.
Схема распределения запросов (Nginx):
1.	Запросы к /images/*: Обрабатываются строго силами Nginx. Nginx напрямую обращается к смонтированной папке images/ на жестком диске и отдает файл клиенту, минуя Python.
2.	Все остальные запросы (/, /list, /upload, /delete): Перенаправляются через proxy_pass на внутренний порт Python-бэкенда (backend:8000).
Спецификация маршрутов бэкенда:
Маршрут (URL)	Метод	Описание функциональности
/	GET	Перенаправляет пользователя на основной интерфейс приложения (/list).
/list	GET	Основной интерфейс (Табы). Принимает GET-параметры. Загружает файл list_template.html.


• Если ?view=upload (или без параметров), отображает drag-and-drop зону загрузки.


• Если ?view=images, делает запрос к БД, формирует таблицу файлов строго по ТЗ и выводит пагинацию (?page=X).
/upload	POST	Загрузка файла. Принимает данные в формате multipart/form-data.


• Проводит валидацию размера (до 5 МБ) и расширения (только .jpg, .jpeg, .png, .gif).


• Сохраняет файл под уникальным UUID-именем на диск и делает запись в таблицу images БД PostgreSQL.


• Возвращает JSON-ответ со статусом и ссылкой.
/delete	GET	Удаление файла. Принимает параметры ?id=X&page=Y.


• Получает имя файла из БД по ID, удаляет физический файл с диска, стирает запись из PostgreSQL.


• Мягко перенаправляет пользователя обратно на ту же страницу пагинации вкладки Images, предотвращая вылет из интерфейса.
Валидация ошибок при загрузке (/upload):
•	При превышении лимита в 5 МБ бэкенд возвращает статус 400 и JSON {"error": "Ошибка! Размер файла больше 5 МБ"}. Фронтенд перехватывает этот ответ и выводит окно alert().
•	При попытке загрузить неподдерживаемый формат (например, .txt, .pdf), бэкенд возвращает статус 400 и JSON {"error": "Ошибка! Тип файла не соответствует файлу картинок"}.

5. Процесс резервного копирования (Backup)
В соответствии с требованиями ТЗ 2.0, в проект внедрена автоматизированная система создания резервных копий базы данных для предотвращения потери информации о загруженных изображениях.
Реализация системы:
Резервное копирование вынесено в отдельный легковесный сервис в docker-compose.yml, который использует стандартную утилиту pg_dump от PostgreSQL.
1.	Периодичность: Резервные копии создаются автоматически по расписанию (скрипт внутри контейнера работает в бесконечном цикле с заданным интервалом, например, раз в сутки или по кастомному таймеру).
2.	Формат и именование: Бэкапы сохраняются в формате SQL-дампа. Имя каждого файла генерируется динамически с указанием точной даты и времени создания, например: backup_2026-06-01_00-00-00.sql.
3.	Безопасность и персистентность: Папка с дампами внутри контейнера смонтирована на физическую папку ./backup/ на хост-машине (компьютере разработчика). Таким образом, даже если контейнеры Docker будут полностью удалены или повреждены, все резервные копии останутся в безопасности на жестком диске.
Процесс восстановления данных из бэкапа:
В случае сбоя или развертывания проекта на новом сервере, восстановить структуру таблиц и данные об изображениях можно одной командой:
Bash
docker compose exec -T db psql -U user_name -d db_name < backup/backup_ИМЯ_ФАЙЛА.sql
(Где user_name и db_name — логин и имя базы данных, указанные в конфигурационных переменных окружения).
